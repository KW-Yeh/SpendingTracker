# 新增 / 編輯帳目 Modal — 設計優化計畫

> 範圍：僅針對 SpendingTracker 的「新增帳目」與「編輯帳目」彈出視窗（modal）。  
> 目標：把原本一頁塞滿的計算機式 modal，重構為更清晰、更符合心流的兩步驟流程。  
> 對應檔案：`direction-a.jsx`（A 方案 — Refined Solo Leveling）

---

## 1. 現況問題

原本的新增/編輯帳目是**單一頁面 + 計算機**的設計。痛點：

| # | 問題 | 影響 |
|---|------|------|
| 1 | 類型 / 必要程度 / 類別 / 描述 / 金額**全擠在一張畫面** | 視覺壓迫、決策路徑不清 |
| 2 | 計算機鍵盤永遠佔據下半部，**即使還沒選類別也是** | 使用者容易誤觸數字、跳過分類 |
| 3 | 類別選擇被擠成小列表 | 點擊區小、emoji 辨識度低 |
| 4 | 沒有步驟感，不知道**還剩幾步** | 容易半途放棄、不確定何時完成 |
| 5 | 確認鍵與類別、描述、金額混在同一視覺層 | 主要 CTA 弱化 |
| 6 | 編輯既有帳目時，跟新增完全一樣，**看不到原本資料的脈絡** | 修改一筆舊帳目時失去 context |

---

## 2. 解法總覽

把單頁拆成 **2 個 Step**，每一步只做一件事：

```
Step 1 ── 選擇細節（What）       Step 2 ── 輸入金額（How much）
─────────────────────────       ─────────────────────────
類型（支出/收入）                 ┌─ Step 1 摘要（emoji + 類別 + 描述 + 必要程度）─┐
必要程度（必要/非必要）           │ 大金額顯示                                     │
類別（5×N emoji 網格）            │ 快速金額 chip（+100 / +500 / +1000 / +5000）   │
描述輸入 + 常用詞 chip            │ 計算機鍵盤                                     │
                                  │ 確認按鈕                                       │
[Next →]                          └─────────────────────────────────────────────┘
```

設計原則：

- **一步一件事**：先決定「這是什麼支出/收入」，再決定「多少錢」
- **狀態保留**：所有 state 由 modal 容器擁有，切換 step 不會 reset
- **進度可見**：右上有 step dots，副標明說「步驟 1 / 2」
- **可往返**：Step 2 左上是 ←（回 Step 1），Step 1 左上是 ✕（關閉 modal）

---

## 3. Step 1 — 選擇細節

### 區塊順序（由上而下）

1. **類型** — `ASeg` segmented control，支出（紅 ↘）/ 收入（綠 ↗）
2. **必要程度** — segmented control，必要 / 非必要
3. **類別** — 5 欄 emoji 網格，依類型動態切換可選類別
4. **描述** — 文字輸入 + 常用詞 chip（早餐 / 午餐 / 晚餐 / 點心 / 飲料 / 咖啡）

### 底部固定 CTA

| 狀態 | 顯示 |
|---|---|
| 還沒輸入金額 | `Next →`（主色 cyan / 紫，按壓進 Step 2） |
| 已輸入金額（編輯模式或從 Step 2 返回） | 上排顯示「已輸入金額 NT$ XXX」+ 按鈕變成「修改金額」 |

> 邏輯：類別預設選好第一個，所以 Next 按鈕**永遠可按**，避免卡住流程。

---

## 4. Step 2 — 輸入金額

### 頂部摘要卡

把 Step 1 選的東西**直接顯示在最上面**，作為 context：

```
┌─────────────────────────────┐
│  🍔  午餐 · 飲食 · 必要      │
└─────────────────────────────┘
```

讓使用者輸入金額時，仍然知道「這筆是在記什麼」。

### 大金額顯示

- 收入：綠色 `+ NT$ 1,200`
- 支出：白色 `− NT$ 220`
- 字級足夠大（44–56px），數字置中

### 快速金額 chip

`+100` / `+500` / `+1000` / `+5000` — 點擊累加到當前金額，常見金額一鍵搞定。

### 計算機鍵盤

- 4 × 4 grid：`7 8 9 ⌫` / `4 5 6 .` / `1 2 3 ✓` / `0 00 ` 等
- 確認鍵 `✓` 用主色實心、其餘為次要灰
- 輸入邏輯由父層 `press(k)` 函式處理，邊界條件：
  - `⌫` 退到剩 1 位時歸零
  - `.` 防止重複插入小數點
  - 從 `0` 開始輸入時直接覆蓋

---

## 5. 互動細節

| 動作 | 行為 |
|---|---|
| 點 Step 1 左上 ✕ | 關閉 modal（state 不保留） |
| 點 Step 2 左上 ← | 回 Step 1，所有 state 保留 |
| 切換「支出 ↔ 收入」 | 類別清單跟著換；若當前 cat 不在新清單則 fallback 到第一個 |
| 切換「必要 ↔ 非必要」 | 純標籤，不影響其他欄位 |
| 點類別 emoji | 立即視覺反饋（光暈 + 主色文字） |
| 點描述 chip | 直接寫入描述輸入框（可再編輯） |
| 點計算機 ✓ | 等同於送出 → 關閉 modal（之後接到 toast/list update） |

## 6. 視覺規範（對應 SURF / palette tokens）

| Token | 用途 |
|---|---|
| `palette.primary` | 主 CTA、active segment 文字、step dot |
| `palette.glow` (rgba) | active segment 背景漸層、類別卡光暈 |
| `SURF.expense` | 支出類型主色（紅） |
| `SURF.income` | 收入類型主色（綠） |
| `SURF.bg1` | Modal 主底（含 8px backdrop blur） |
| `SURF.bg2` | Input 底色 |
| `SURF.line` / `lineSoft` | 一般 / 弱化邊框 |
| `SURF.text` / `textDim` / `textMuted` | 三層文字權重 |

圓角規則：

- Modal：`24px`（僅上方兩角）
- Input / segment / 類別卡：`12px`
- Chip：`99px` (pill)

| 元件 | 職責 |
|---|---|
| `AEditModal` | 容器：擁有所有 state、切換 step、頭部 step indicator |

| `AEditStep1` | Step 1 內容：類型 / 必要 / 類別 / 描述 |
| `AEditStep2` | Step 2 內容：摘要卡 / 大金額 / quick chips / 計算機 |
